Replace numeric sequences with the token `number`.

Applies to:
- order numbers
- ticket numbers

- dates if represented numerically

- "$49.99" -> "number"

### 8) URL Normalization
All URLs MUST be replaced with the token `url`.

Examples:
- https://example.com/deal -> url
- www.example.com -> url

Do not keep raw URLs in normalized training text.

---

### 9) Flags
Always include these flags explicitly:
- has_attachment
- is_reply
- is_forward
- has_unsubscribe

Render exactly as:
flags: has_attachment={true|false} is_reply={true|false} is_forward={true|false} has_unsubscribe={true|false}

Definitions:
- `has_attachment=true` if the message has one or more attachments
- `is_reply=true` if the message is part of a reply flow
- `is_forward=true` if the message is clearly forwarded
- `has_unsubscribe=true` if unsubscribe capability is detected, including
  List-Unsubscribe or clear unsubscribe content

Do not omit any flag, even when false.

---

### 10) Gmail Labels
Do NOT include raw Gmail labels in the normalized classifier input text.

Do NOT output lines like:
- gmail_label: CATEGORY_PROMOTIONS,UNREAD,INBOX
- gmail_label: IMPORTANT

Reason:
- this can create label leakage
- the classifier may learn Gmail's answer instead of learning email patterns

Gmail labels MAY be used only in:
- bootstrap labeling / auto-teach logic
- metadata stored outside the TF-IDF input
- audits and review tools

If Gmail metadata is needed experimentally, it MUST be added as a separate,
controlled feature flag system and never as a raw label dump.

---

### 11) Stable Structure
Keep the output field order exactly the same for all samples:

1. from
2. domain
3. recipient_flags
4. subject
5. flags
6. body

Do not reorder fields dynamically.

Do not omit empty fields except body content may be empty after cleaning.

---

### 12) No Extra Metadata in Training Text
Do NOT include these in normalized classifier input:
- message-id
- DKIM headers
- received headers
- MIME boundaries
- raw gmail labels
- raw timestamps
- raw thread IDs
- mailbox folder names
- internal database IDs
- auth results
- tracking hashes
- long random tokens

This metadata may be stored separately, but not in the TF-IDF input text.

## Training Data Guidance
Each labeled training row MUST contain:
- normalized flat text
- final label
- normalization_version
- label_source

Recommended label_source values:
- manual
- gmail_bootstrap
- rule_bootstrap
- corrected_prediction

Training SHOULD prefer:
- manual labels
- corrected predictions

Bootstrap labels may be used, but they should be distinguishable by source.

---

## Inference Guidance
At inference time:
- apply the exact same normalization
- do not bypass normalization for "simple" emails
- do not feed raw bodies directly into the vectorizer

The normalized text generated during inference MUST match the training format.

---

## Model Compatibility
Every trained model MUST store:
- normalization_version
- label set
- threshold config
- training timestamp
- training sample counts

If normalization rules change:
- increment `normalization_version`
- retrain the model
- do not silently mix incompatible normalized data with older models

---

## Acceptance Criteria
A normalization implementation is acceptable only if:
- it produces deterministic output for the same input
- it lowercases consistently
- it removes HTML noise
- it replaces URLs with `url`
- it replaces numbers with `number`
- it buckets recipient_count correctly
- it excludes raw Gmail labels from classifier input
- it preserves the exact field order
- it uses the same rules in training and inference
